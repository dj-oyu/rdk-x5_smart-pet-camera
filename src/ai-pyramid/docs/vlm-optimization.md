# VLM 呼び出し最適化レポート

Qwen3-VL-2B on axllm (AX8850) の実機ベンチマークに基づく、VLM 呼び出し実装の評価と改善提案。

## 推論バックエンド変遷

| 項目 | StackFlow (旧) | axllm (現在) |
|------|---------------|-------------|
| VLMテキスト速度 | 3.9 tok/s | **9.1 tok/s** |
| VLM画像+テキスト速度 | 3.9 tok/s | **9.2 tok/s** |
| TTFT (テキスト) | 400ms | **316ms** |
| TTFT (画像) | 750ms | **352ms** |
| コンテキスト長 | 1,152 tokens | **3,584 tokens** |
| CMM使用量 | 3,582MB | 2,887MB |
| API | StackFlow独自 (FastAPI→TCP) | OpenAI互換 (axllm serve) |

モデル: `Qwen3-VL-2B-Instruct-GPTQ-Int4-C256-P3584-CTX4095`

## 実機ベンチマーク結果 (StackFlow時代)

> 以下はStackFlowでの計測値。axllm移行後は速度が約2.4倍に向上。

### プロンプト長の影響（画像付き、3回平均）

| プロンプト | 長さ | 応答時間 |
|-----------|------|---------|
| 短い (`Cat? JSON:...`) | 61 chars | 2.74s |
| VLM_PROMPT（現在の実装） | 258 chars | 3.43s |

差分: **~0.7s**。現在のプロンプト長は許容範囲内。

### 画像サイズの影響（実写ペットカメラ画像）

| 画像 | リクエストサイズ | 応答時間 |
|------|----------------|---------|
| オリジナル 844×644 (117KB JPEG) | 155KB | 4.75s |
| リサイズ 384×384 (15KB JPEG) | 32KB | 5.31s |

ビジョンエンコーダは内部で 384×384 にリサイズするため、**推論レイテンシへの影響はほぼゼロ**。

## 現在の実装の評価

### 良い点

| 項目 | 評価 |
|------|------|
| `temperature: 0.1` | JSON応答の安定性に最適 |
| base64 インライン送信 | localhost なので最速 |
| 逐次処理 (1 req/image) | NPU は排他リソース。並列化は不要 |
| `max_tokens: 128` | デフォルト 1000 より大幅に節約 |
| `stream: false` | バッチ処理なのでストリーミング不要 |
| リトライ 1 回 | axllm の一時エラー対策として妥当 |
| markdown fence パース | モデルが ````json` で囲む場合のフォールバック |
| `--vlm-model` CLI引数 | モデルID変更に柔軟に対応 |

### 改善が必要な点

#### 1. [中] `summarize_day` のキャプション数制限

**問題:**
```rust
let observations = captions.join("\n- ");
let user_text = format!("Observations:\n- {observations}\n\n{DAY_SUMMARY_PROMPT}");
// max_tokens: 256
```

axllm移行でコンテキスト長が 1,152 → 3,584 トークンに拡張されたため、
StackFlow時代ほど切迫していない。ただし1日50枚 × 平均60文字 = 3000文字以上の
プロンプトになると、3,584トークン上限に近づく可能性がある。

**提案:** キャプションを最新 N 件に制限（安全マージン）。

```rust
// 例: 最新30件に制限 (axllmの3584トークンなら十分)
let recent = captions.iter().rev().take(30).rev();
let observations = recent.cloned().collect::<Vec<_>>().join("\n- ");
```

#### 2. [低] 画像の事前リサイズ

推論速度への影響はほぼゼロだが、以下のメリットがある:

- **メモリ効率**: 117KB → 15KB (リクエストボディが 1/5)
- **ネットワーク帯域**: localhost でも JSON パース負荷が減る

ただし、Rust で画像リサイズを行うには `image` crate の追加が必要で、
バイナリサイズが増加する（~数百KB）。**優先度は低い**。

#### 3. [低] `max_tokens` の最適化

現在 `max_tokens: 128` だが、VLM の JSON 応答は通常 50-80 トークン程度。
axllmではリクエストごとに `max_tokens` を動的に制御できるため、
用途に応じた最適化が可能。現状でも問題なし。

## axllm移行で解消された制約

| 旧制約 (StackFlow) | axllm での状況 |
|-------------------|---------------|
| コンテキスト長 1,152 トークン | **3,584 トークン** に拡張 |
| `max_tokens` がセッション単位で固定 | リクエストごとに動的制御可能 |
| 会話履歴なし (`_truncate_history`) | 標準のOpenAI API準拠 |
| `timeout: 30s` がギリギリ (128tok × 264ms = 33.8s) | **128tok × 109ms = 14s** で余裕 |
| Python仲介レイヤーのオーバーヘッド | NPU直接呼び出し |

## 変更不要な項目

| 項目 | 理由 |
|------|------|
| プロンプトの短縮 | 0.7s の差であり、応答品質を犠牲にする価値がない |
| 画像解像度 | 推論速度への影響はゼロ（内部で 384×384 にリサイズ） |
| temperature | 速度に影響しない（計測済み: 0.0〜1.0 で同一速度） |

## 推奨アクション（優先度順）

1. `summarize_day` のキャプション数を制限する（コンテキスト超過回避）
2. （任意）画像リサイズの実装（メモリ効率改善、速度影響なし）
